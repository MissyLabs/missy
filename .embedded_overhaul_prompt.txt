CURRENT PRIMARY FOCUS

complete tool usage and tool intelligence overhaul.

For the default tool-focused run, this means a real, end-to-end overhaul of
how Missy discovers, creates, stores, tests, benchmarks, routes, and enables
tools.

The system must learn from frequent user requests and repeated manual
workflows. When a pattern is common enough and safe enough, Missy should be
able to propose or create a structured tool candidate instead of repeatedly
solving the same task ad hoc. Tool creation must be policy-gated, auditable,
reviewable, testable, reversible, and never allowed to bypass security.

Required tool-intelligence capabilities include:

- frequent-request and repeated-workflow detection
- safe tool-candidate generation from observed requests
- clean tool storage with metadata, provenance, permissions, schemas, tests,
  examples, versions, ownership, and lifecycle state
- clear separation between proposed, experimental, benchmarked, approved,
  enabled, deprecated, and disabled tools
- provider-aware tool schemas and compatibility metadata
- benchmark suites that run tool tasks against providers such as OpenAI,
  Ollama, Anthropic where present, and local/mock providers
- benchmark scoring for correctness, latency, cost, reliability, safety,
  schema adherence, tool-call quality, and failure behavior
- provider-specific enablement flags based on benchmark results
- fallback behavior when a provider is weak at a tool or tool family
- CLI and diagnostics for listing candidates, reviewing tools, running
  benchmarks, comparing providers, and enabling/disabling provider access
- audit trails for tool synthesis, benchmark execution, approval, enablement,
  denial, and rollback

The overhaul should be architectural, not cosmetic. Prefer clear internal
interfaces, explicit lifecycle management, robust error handling, policy-gated
actions, structured audit events, and tests that prove behavior. If
PRIMARY_FOCUS is changed for a later run, apply this same standard to that
subsystem instead of forcing tool work.

SECURITY REQUIREMENTS

- Default-deny mindset for privileged actions.
- Tool creation, tool execution, Discord, and scheduled execution must be safe
  by default.
- Secrets must come from environment or protected local files and be redacted.
- Network, filesystem, shell, plugin, provider, and Discord actions must go
  through policy where applicable.
- Dangerous actions need approval or explicit policy.
- Audit logs must record privileged actions and denials.
- Defend against prompt injection, malicious generated tools, tool supply-chain
  abuse, data exfiltration, SSRF, scheduler abuse, secrets leakage, Discord
  spoofing, bot loops, impersonation, unsafe attachments, confused channel
  routing, benchmark gaming, and unsafe provider-specific tool enablement.
